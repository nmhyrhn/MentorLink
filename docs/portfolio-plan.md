# MentorLink 백엔드 프로젝트 기획서

## 1. 프로젝트를 시작하게 된 배경

이번 프로젝트는 백엔드 채용 공고에서 반복적으로 요구되는 핵심 역량을 포트폴리오 형태로 증명하기 위해 기획했습니다. 특히 다음 역량을 실제 서비스 흐름 안에서 보여주는 것이 목적입니다.

- Java 기반 백엔드 개발 경험
- Spring Boot 기반 REST API 설계 역량
- JPA 기반 데이터 모델링 및 ERD 설계 능력
- 승인 시 세션 생성과 같은 트랜잭션 처리 설계
- AWS EC2, Docker, Nginx를 활용한 배포 경험
- 실제 서비스와 유사한 상태 전이형 워크플로우 설계 경험

채용 서비스와 동일한 도메인을 그대로 복제하기보다, 구조적으로 유사한 워크플로우를 갖는 매칭형 서비스를 설계하는 것이 포트폴리오 설명에 더 적합하다고 판단했습니다. 그래서 `지원 -> 승인 -> 인터뷰 -> 평가` 흐름을 `멘토링 신청 -> 승인 -> 세션 진행 -> 리뷰`로 재구성한 `MentorLink`를 주제로 선택했습니다.

## 2. 프로젝트 목표

MentorLink를 통해 보여주고자 하는 백엔드 역량은 다음과 같습니다.

- 상태 전이가 분명한 도메인을 기반으로 엔티티와 API를 설계하는 능력
- 인증, 인가, 비즈니스 로직, 영속성 계층을 분리하는 계층형 아키텍처 설계 능력
- Application 중심의 워크플로우 설계와 트랜잭션 처리 능력
- 프론트엔드와 연결 가능한 REST API 서버 구현 능력
- Docker 기반 컨테이너 실행과 AWS 배포 경험

즉, 단순 CRUD 프로젝트가 아니라 "업무 흐름이 있는 서비스형 백엔드"를 구현하는 것이 핵심 목표입니다.

## 3. 서비스 개요

MentorLink의 전체 흐름은 다음과 같습니다.

멘토 등록  
-> 멘토 목록 조회  
-> 멘토링 신청  
-> 신청 승인 또는 거절  
-> 세션 생성  
-> 세션 진행  
-> 리뷰 작성

멘티는 멘토 목록에서 원하는 멘토를 탐색한 뒤 멘토링을 신청합니다. 멘토가 해당 신청을 승인하면 세션이 자동으로 생성됩니다. 세션이 종료되면 멘티는 리뷰를 남기고, 하나의 매칭 사이클이 마무리됩니다.

이 구조는 "신청, 승인, 실행, 평가"라는 서비스형 워크플로우를 명확하게 보여주며, 포트폴리오 면접에서 도메인 설계 이유를 설명하기 좋습니다.

## 4. 핵심 기능 정의

### 회원가입

사용자가 멘토 또는 멘티 역할을 선택하여 계정을 생성합니다. 인증 기반 서비스의 시작점이며, 이후 역할별 인가 처리의 기준이 됩니다.

### 로그인

Spring Security와 JWT 기반으로 인증을 수행합니다. 로그인 성공 시 액세스 토큰을 발급하고, 이후 API는 토큰 기반 인증으로 동작합니다.

### 멘토 프로필 등록

멘토 권한 사용자가 자신의 전문 분야, 경력, 소개를 입력해 멘토 프로필을 생성합니다. `User`와 `MentorProfile`를 분리해 계정 정보와 비즈니스 프로필 정보를 구분합니다.

### 멘토 목록 조회

멘티가 멘토를 탐색하기 위한 기능입니다. 공개 조회 API로 설계해 로그인 전에도 서비스 구조를 보여줄 수 있습니다.

### 멘토링 신청

멘티가 특정 멘토에게 메시지와 함께 신청을 생성합니다. 이때 `Application` 엔티티가 생성되며 상태는 `PENDING`으로 시작합니다.

### 신청 승인 / 거절

멘토가 자신에게 들어온 신청을 승인하거나 거절합니다. 승인 시에는 상태 변경뿐 아니라 세션 생성까지 함께 일어나므로 핵심 트랜잭션 포인트가 됩니다.

### 세션 관리

승인된 신청에 대해 생성된 세션을 조회하고 완료 처리합니다. 세션 완료 시 Application 상태도 `COMPLETED`로 전이되어 전체 흐름이 닫히게 됩니다.

### 리뷰 작성

세션 종료 후 멘티가 리뷰를 남깁니다. 세션당 하나의 리뷰만 생성 가능하도록 1:1 관계를 설정해 서비스 규칙을 명확히 반영합니다.

## 5. 기술 스택 선정 이유

### Backend

- Java  
  객체지향 설계와 계층형 아키텍처를 보여주기에 적합하고, 실무 백엔드 채용에서 여전히 핵심 언어입니다.

- Spring Boot  
  REST API 개발, 의존성 관리, 설정 자동화, 테스트 지원 등 백엔드 개발 생산성이 높고 실무 친화적입니다.

- Spring Data JPA  
  엔티티 중심 설계와 연관관계 매핑, 트랜잭션 기반 데이터 접근을 설명하기 좋습니다.

- Spring Security (JWT 인증)  
  단순 로그인 구현을 넘어서 인증/인가 구조를 설계했다는 점을 보여줄 수 있습니다.

- MySQL  
  실무에서 널리 쓰이는 관계형 데이터베이스이며, 매칭형 서비스의 관계 중심 데이터를 표현하기 적합합니다.

### Infra

- AWS EC2  
  실제 서버 배포 경험을 설명하기 좋고, 포트폴리오 면접에서 인프라 구성까지 이야기할 수 있습니다.

- Docker  
  개발 환경과 배포 환경을 일관성 있게 유지할 수 있고, 컨테이너 기반 운영 경험을 보여줄 수 있습니다.

- Nginx  
  Reverse Proxy, 정적/동적 라우팅 분리, 프론트엔드와 백엔드 연결 구조를 설명하기 좋습니다.

### Dev

- GitHub  
  버전 관리와 협업 기본기를 보여줄 수 있습니다.

- Swagger  
  API 문서화를 통해 백엔드 결과물을 명확하게 전달할 수 있습니다.

## 6. 데이터 모델 설계 방향

핵심 엔티티는 다음과 같습니다.

- `User`
- `MentorProfile`
- `Application`
- `MentoringSession`
- `Review`

### Application 엔티티가 핵심인 이유

이 프로젝트의 핵심은 단순한 멘토 조회가 아니라, "신청이 생성되고 승인되며 세션이 만들어지고 완료되는 상태 전이"입니다. 그 중심에는 항상 `Application`이 있습니다.

`Application`은 다음 역할을 수행합니다.

- 멘토와 멘티를 연결하는 매칭 기준점
- 현재 워크플로우 상태(`PENDING`, `APPROVED`, `REJECTED`, `COMPLETED`)를 보관
- 세션 생성 여부를 판단하는 기준 엔티티
- 트랜잭션 처리의 시작점

즉, 이 프로젝트는 "세션 중심 서비스"가 아니라 "신청 중심 워크플로우 서비스"로 설계되었습니다.

### 관계 구조

- 멘토 1 : N Application
- 멘티 1 : N Application
- Application 1 : 1 Session
- Session 1 : 1 Review

이 구조는 상태 전이형 서비스에서 데이터 흐름이 자연스럽게 이어지도록 설계한 결과입니다.

## 7. 트랜잭션 설계

가장 중요한 트랜잭션 포인트는 멘토링 신청 승인입니다.

승인 로직에서는 다음 두 작업이 함께 수행됩니다.

1. `Application` 상태를 `APPROVED`로 변경
2. 해당 신청에 대응하는 `Session` 생성

이 두 작업은 반드시 하나의 트랜잭션으로 처리되어야 합니다.

그 이유는 다음과 같습니다.

- 상태만 승인되고 세션이 생성되지 않으면 데이터 정합성이 깨집니다.
- 세션만 생성되고 상태가 승인으로 바뀌지 않으면 비즈니스 규칙이 깨집니다.
- 사용자는 "승인 완료"를 하나의 원자적 작업으로 인식하기 때문에 중간 실패 상태가 없어야 합니다.

따라서 승인 API는 `ApplicationService`에서 트랜잭션으로 묶어 처리하도록 설계했습니다.

## 8. 시스템 아키텍처

기본 구조는 다음과 같습니다.

Client  
↓  
Nginx  
↓  
Spring Boot API Server  
↓  
MySQL

Docker 기반으로 보면 다음과 같이 설명할 수 있습니다.

- `frontend` 컨테이너: Next.js 애플리케이션
- `nginx` 컨테이너: Reverse Proxy
- `backend` 컨테이너: Spring Boot API 서버
- `mysql` 컨테이너: 데이터 저장소

Nginx는 `/` 요청은 프론트엔드로, `/api` 요청은 Spring Boot 서버로 라우팅합니다. 이 구조는 Next.js와 API 서버를 자연스럽게 연결할 수 있는 실무형 배포 방식입니다.

## 9. API 설계 방향

REST API 중심으로 설계했습니다.

예시:

- `POST /auth/signup`
- `POST /auth/login`
- `GET /auth/me`
- `GET /mentors`
- `GET /mentors/{id}`
- `POST /mentors/profile`
- `POST /applications`
- `PATCH /applications/{id}/approve`
- `PATCH /applications/{id}/reject`
- `GET /applications/me/received`
- `GET /sessions/me`
- `PATCH /sessions/{id}/complete`
- `POST /reviews`

설계 기준은 다음과 같습니다.

- 리소스 중심 URI 사용
- 상태 변경은 `PATCH` 사용
- 로그인 사용자는 JWT로 식별하고, 요청 바디에 사용자 ID를 직접 넘기지 않음
- 프론트엔드가 바로 소비할 수 있도록 JSON 응답 구조를 단순화

## 10. 배포 전략

배포 흐름은 다음과 같습니다.

GitHub  
-> Docker 이미지 생성  
-> AWS EC2 배포  
-> Nginx Reverse Proxy 설정  
-> Spring Boot 실행

상세 전략:

- GitHub Actions 또는 수동 빌드로 Docker 이미지를 생성
- EC2 서버에 Docker와 Docker Compose 설치
- `frontend`, `backend`, `mysql`, `nginx` 컨테이너 실행
- Nginx가 외부 트래픽을 받고 `/api`는 Spring Boot, `/`는 Next.js로 전달

이 구조는 향후 Next.js 프론트엔드와 연결 가능한 API 서버 구조를 그대로 유지합니다. 프론트와 백엔드를 분리 배포하더라도 Reverse Proxy 기반으로 쉽게 확장할 수 있습니다.

## 12. 포트폴리오에서 강조할 포인트

### 데이터 모델링 설계

단순 게시판이 아니라 워크플로우가 있는 서비스이기 때문에, 엔티티 간 관계와 상태 전이가 자연스럽게 이어지도록 설계했다는 점을 강조할 수 있습니다.

### Application 중심 워크플로우 설계

서비스 핵심을 `Application`에 두고, 승인 시 세션 생성까지 연결되는 흐름을 설계했다는 점이 가장 중요한 차별점입니다.

### 트랜잭션 처리

승인 시 상태 변경과 세션 생성이 함께 일어나도록 묶어 데이터 정합성을 보장한 점을 설명할 수 있습니다.

### REST API 설계

리소스 중심 URI, JWT 기반 인증, 역할 기반 인가, 상태 변경 API 분리 등 실무형 API 설계를 했다는 점을 보여줄 수 있습니다.

### 클라우드 배포 경험

EC2, Docker, Nginx를 조합해 프론트엔드와 백엔드가 분리된 서비스 구조를 직접 배포한 경험으로 설명할 수 있습니다.

## 면접에서 이렇게 설명하면 좋습니다

"이 프로젝트는 단순 CRUD보다 실제 서비스와 유사한 워크플로우를 보여주기 위해 설계했습니다. 핵심은 `Application` 엔티티를 중심으로 신청, 승인, 세션 생성, 리뷰 작성까지 상태 전이를 명확히 모델링한 점입니다. 특히 승인 시 `Application` 상태 변경과 `Session` 생성을 하나의 트랜잭션으로 처리해 데이터 정합성을 보장했고, JWT 기반 인증과 Docker, Nginx, EC2 배포 구조까지 포함해 백엔드 개발자가 실무에서 다루는 흐름을 전체적으로 담으려 했습니다."
