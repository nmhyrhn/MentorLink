# MentorLink QA 문서

## 1. 테스트 목적

- 멘토-멘티 매칭 서비스의 핵심 흐름이 정상 동작하는지 확인
- 인증, 예약, 승인, 세션, 리뷰 흐름의 예외 케이스를 점검
- 포트폴리오 제출 전 배포 직전 상태에서 확인된 리스크를 정리

## 2. 테스트 환경

- Frontend: Next.js
- Backend: Spring Boot
- DB: MySQL 8
- Infra: Docker Compose, Nginx
- 실행 주소: `http://localhost`
- Swagger: `http://localhost/api/swagger-ui.html`

## 4. 확인한 데이터 흐름

### 회원가입

1. 이메일 인증 코드 발송
2. 이메일 인증 코드 검증
3. 회원가입
4. JWT 발급

### 멘토링 신청

1. 멘토 상세 조회
2. 예약 가능 슬롯 조회
3. 날짜, 시작 시간, 상담 시간을 선택
4. 연락처와 상담 메시지 입력
5. 신청 생성

### 멘토 승인

1. 받은 신청 목록 조회
2. 승인 또는 거절
3. 승인 시 세션 자동 생성

### 세션 종료와 리뷰

1. 세션 완료 처리
2. 멘티가 후기 작성
3. 후기 데이터가 멘토 프로필에 반영

## 5. 보완이 필요한 항목

### 높음

- JWT 시크릿이 잘못 설정되어도 하드코딩된 기본 키로 대체됩니다.
- 백엔드 재시작 시 현재 구현은 목업 데이터를 다시 밀어 넣습니다.

### 중간

- Swagger가 모든 환경에서 공개되어 있습니다.
- SMTP 실패 시 개발용 인증코드 fallback 응답을 허용하고 있습니다.
- 현재 메일 디버그 로그가 운영 환경까지 켜질 수 있습니다.

### 낮음

- 로그인/이메일 인증 요청에 대한 rate limiting이 없습니다.
- 예약 규칙과 전문 분야가 CSV 저장 방식이라 확장성은 제한적입니다.

## 6. 권장 개선안

### 배포 전 필수

- 운영 환경에서는 `APP_JWT_SECRET` 누락 시 서버가 기동 실패하도록 변경
- `DataInitializer`를 `dev` 또는 `demo` 프로필에서만 실행
- `APP_EMAIL_DEBUG_FALLBACK=false` 기본화
- `MAIL_DEBUG=false`, `MAIL_LOG_LEVEL=INFO` 또는 `WARN`로 하향
- Swagger는 `dev` 프로필에서만 공개

### 다음 단계 개선

- Redis 기반 인증 코드 만료/재시도 제한
- 로그인 실패 횟수 제한 또는 rate limiting
- availability / expertise 정규화 테이블 분리
- Refresh Token 또는 재로그인 정책 분리
